# Final Report: LLM-Based Support Ticket Auto-Tagging

**Task 5 — AI/ML Engineering Advanced Internship, DevelopersHub Corporation**

## 1. Problem Statement

Support teams receive a high volume of free-text tickets that must be routed
to the right queue quickly. Manually tagging each ticket doesn't scale, and
traditional supervised classifiers require large amounts of labeled training
data per category. This project builds an **LLM-based auto-tagging system**
that classifies tickets into 8 categories using two prompting strategies —
**zero-shot** and **few-shot** — that need little to no labeled data, and
compares which strategy works better for this task.

## 2. Dataset

`data/support_tickets.csv` — 8 categories (Billing and Payments, Technical
Issue, Account Access, Product Inquiry, Cancellation Request, Shipping and
Delivery, Bug Report, Feature Request), generated from realistic phrasing
templates with randomized slot-fillers (product names, order IDs, amounts).
After deduplication this yields ~258 unique tickets, split 70/30 into
train/test (stratified). The train split is unused by the classifiers
themselves (both are prompt-based, not fine-tuned) — it exists so a fair
"held-out" evaluation set can be defined and so the hand-picked few-shot
examples in `src/config.py` can be verified as non-overlapping with the test
tickets.

*Why synthetic-but-templated data instead of a public dataset?* Public
ticket datasets are either gated behind logins, tiny, or too noisy to
evaluate reproducibly in a fixed internship timeline. The template approach
still produces realistic difficulty: categories share vocabulary (e.g.
"Cancellation Request" and "Billing and Payments" both mention charges),
which is exactly what makes zero-shot vs. few-shot comparison meaningful.

## 3. Methodology

### 3.1 Zero-shot classification
Model: `facebook/bart-large-mnli` via HuggingFace's `zero-shot-classification`
pipeline. Each candidate category becomes an NLI hypothesis ("This support
ticket is about *Billing and Payments*.") and the model's entailment
probability across all 8 hypotheses is normalized into a single ranked
distribution — no examples, no training.

### 3.2 Few-shot classification
Model: `google/flan-t5-base`. A single prompt is constructed containing 1–2
labeled example tickets per category (the "shots"), followed by the new
ticket. Rather than freely generating a category name (hard to parse
reliably and gives no confidence score), we **score each candidate
category's log-likelihood as a forced decoder continuation**, length-
normalize it, and softmax across all 8 candidates. This is the same
log-likelihood-scoring technique used in the GPT-3 paper's few-shot
evaluations, and it gives few-shot classification a ranked probability
output directly comparable to the zero-shot pipeline's.

### 3.3 Top-3 prediction
Both classifiers naturally output a full ranked distribution over all 8
categories; we take the top 3 as the "probable categories" for each ticket.

### 3.4 Evaluation
- **Accuracy**, **macro precision/recall/F1** on top-1 predictions
- **Top-3 accuracy**: fraction of tickets where the true category appears
  anywhere in the top-3 ranked output
- **Confusion matrices** and **per-category F1** to see which categories
  each method struggles with
- **Average inference time per ticket**, since latency matters for a
  production tagging system

All metrics and raw predictions are saved to `results/metrics.json` and
`results/predictions.csv` by `src/evaluate.py`; visualizations are in
`results/figures/` (generated by `src/visualize.py`).

## 5. Expected Findings & Discussion

Based on the design of the two methods, here is what to look for once you
run the pipeline, and why:

- **Zero-shot tends to do well when category names are self-descriptive.**
  NLI models are pretrained on huge amounts of natural language entailment
  data, so a hypothesis like "This ticket is about Shipping and Delivery"
  is easy to score correctly even with zero examples, because the category
  name itself carries most of the needed signal.
- **Few-shot should help most on ambiguous or overlapping categories**
  (e.g. "Cancellation Request" vs. "Billing and Payments" — both mention
  charges/refunds). The in-context examples give the model concrete
  boundary cases it can pattern-match against, which a category name alone
  can't convey.
- **Top-3 accuracy should be noticeably higher than top-1 accuracy for
  both methods** — most classification errors in this kind of task are
  "near misses" between semantically adjacent categories (Bug Report vs.
  Technical Issue, for example), and both usually land in each other's
  top-3.
- **Zero-shot is likely faster per ticket** than the few-shot scoring
  approach, since few-shot requires one forward pass *per candidate label*
  to compute log-likelihoods (8 forward passes vs. effectively 1 batched
  NLI pass), which is a real deployment cost/latency tradeoff worth noting
  even when few-shot wins on accuracy.

## 6. Limitations

- The dataset is template-generated, not real production ticket data —
  results won't fully capture the messiness (typos, mixed languages,
  very short/very long tickets) of real support queues.
- Few-shot examples were hand-picked rather than retrieved dynamically
  (e.g. via embedding similarity to the query ticket); a retrieval-
  augmented few-shot approach would likely improve results further.
- `flan-t5-base` is a small model chosen for CPU-friendly reproducibility;
  a larger instruction-tuned LLM would likely widen or narrow the gap with
  zero-shot depending on category difficulty.
- Categories in this project are mutually exclusive by design; real
  ticketing systems often need multi-label tagging (a ticket can be both
  a "Bug Report" and a "Technical Issue"), which both pipelines support
  in principle (the underlying scores are per-label) but which isn't
  evaluated here.

## 7. Conclusion

This project demonstrates a complete, trainable-data-free pipeline for
LLM-based support ticket classification: a zero-shot NLI baseline, a
few-shot log-likelihood-scoring classifier, ranked top-3 predictions for
both, and a full evaluation/visualization/deployment stack (Streamlit).
The comparison framework built here (`src/evaluate.py`, `src/visualize.py`)
is reusable for any future dataset or model swap — only `src/config.py`
needs to change to point at new data, categories, or models.
